# Provisum — Test Runner Guide for Claude Code

This document contains step-by-step instructions for Claude Code to execute the Provisum test suite, triage failures, and produce a structured report. Run these instructions from the project root: `AI Role Mapping Tool/airm/`

---

## Prerequisites

Before running any tests, verify the environment:

```bash
# Confirm you are in the correct project root
ls package.json playwright.config.ts vitest.config.ts

# Confirm pnpm is available
pnpm --version

# Confirm Node version (18+ required)
node --version
```

---

## Step 1 — Install Dependencies

```bash
pnpm install
```

If Playwright browsers are not installed:

```bash
pnpm playwright install --with-deps chromium
```

---

## Step 2 — Environment Setup

The test suite requires environment variables. Check for an env file:

```bash
ls .env* 2>/dev/null | head -5
```

If no `.env.local` exists, copy the template:

```bash
cp .env.example .env.local
```

Then fill in the required values. The minimum required for tests:

```env
# Supabase Postgres — pooled connection string (port 6543)
DATABASE_URL=postgresql://postgres.<project-ref>:<password>@aws-0-<region>.pooler.supabase.com:6543/postgres

# Supabase Auth
NEXT_PUBLIC_SUPABASE_URL=https://<project-ref>.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=<your-anon-key>
SUPABASE_SERVICE_ROLE_KEY=<your-service-role-key>

# Anthropic (required for Lumen AI features)
ANTHROPIC_API_KEY=sk-ant-v4-...

# Encryption key for settings at rest (optional in dev, required in prod)
ENCRYPTION_KEY=<base64-encoded-32-byte-key>

# App URL (used in global-setup.ts and email links)
NEXT_PUBLIC_APP_URL=http://localhost:3000
```

> **Note:** `.env.local` is gitignored. Never commit it. For CI, set these as repository secrets.

## Step 3 — Run Vitest Unit Tests

These tests have no network dependencies and run in under 15 seconds.

```bash
pnpm test
```

Expected output:

```
✓ __tests__/middleware.test.ts          (12 tests)
✓ __tests__/lib/auth.test.ts            (18 tests)
✓ __tests__/lib/settings.test.ts        (8 tests)
✓ __tests__/lib/strapline.test.ts       (6 tests)
```

**If tests fail:**

1. Run with verbose output: `pnpm vitest run --reporter=verbose`
2. Check for import path errors first — a `Cannot find module '@/lib/...'` error means a source file was moved or the `@` alias is misconfigured in `vitest.config.ts`.
3. Auth hierarchy failures are highest priority — they indicate a regression in `lib/auth.ts` role ordering or `requireRole()` behavior.

**Coverage report:**

```bash
pnpm vitest run --coverage
```

Open `coverage/index.html` to inspect which lines are uncovered.

Coverage targets:
- `lib/auth.ts` ≥ 90%
- `lib/scope.ts` ≥ 85%
- `lib/password-policy.ts` ≥ 95%
- `middleware.ts` ≥ 80%

## Step 4 — Verify Auth State Files Before E2E

The E2E suite uses pre-saved session cookies generated by `global-setup.ts`. If the `.auth/` directory is missing or stale, all role-based tests will fail with redirect-to-login errors.

Check that auth state files exist:

```bash
ls e2e/.auth/
```

Expected output:

```
demo.admin.json
demo.approver.json
demo.coordinator.json
demo.mapper.finance.json
demo.viewer.json
sysadmin.json
```

If any files are missing, the `global-setup.ts` did not complete cleanly. Re-run setup by deleting the `.auth/` directory and running the full suite once:

```bash
rm -rf e2e/.auth/
pnpm test:e2e --project=chromium
```

The global setup will re-authenticate all 6 users via `POST /api/auth/login` and save their cookies.

**If login calls fail in global-setup.ts:**
Check that the dev server is reachable and demo users are seeded:

```bash
# Seed demo data (required once per fresh database)
pnpm db:seed --demo=standard
```

Let Playwright manage the dev server automatically (uses the `webServer` config):

**Run specific suites:**


```bash

```

## Step 6 — Triage Failures

### Category A: Auth state stale or missing

**Symptom:** Tests that call `login(page, "demo.admin")` immediately redirect to `/login` instead of the expected page. URL in failure screenshots shows `/login`.

**Fix:** Delete `.auth/` and re-run the full suite to regenerate all six cookie files. See Step 4.

```bash
rm -rf e2e/.auth/ && pnpm test:e2e
```

---

### Category B: Empty state — seed data missing

**Symptom:** Page loads correctly (no redirect) but assertions like `expect(count).toBeGreaterThan(0)` fail because tables are empty. The dashboard shows "No personas found" or similar zero-state UI.

**Fix:** Re-seed the database:

```bash
pnpm db:seed --demo=standard
```

Then re-run the failing spec:

```bash
pnpm playwright test e2e/full-workflow.spec.ts
```

---

### Category C: Timing / flaky

**Symptom:** A test fails on the first attempt but passes on retry. The Playwright HTML report shows a ⚠ flaky badge.

**Known flaky test:** `role-access-matrix.spec.ts` → `admin role cannot access /admin`

**Root cause:** The `requireRole()` server-side redirect occasionally resolves after Playwright's `waitForURL` fires. The test passes on retry with no code change.

**Fix:** If the same test flakes on more than 2 consecutive runs, investigate for a real timing regression. Otherwise, `retries: 1` in `playwright.config.ts` absorbs the flake. Do not mark it `test.skip()`.

---

### Category D: Route not found (404)

**Symptom:** A test navigating to a new route (e.g., `/workstream`, `/risk-analysis`) gets a 404 page or is redirected. The page heading assertion fails because the route doesn't exist yet.

**Fix:** Check whether the route directory exists in `app/`:

```bash
ls app/workstream/
ls app/risk-analysis/
```

If the directory is missing, the feature was not yet deployed to the branch being tested. Mark the affected tests `test.skip()` with a comment referencing the sprint:

```ts
test.skip("workstream page loads", async ({ page }) => {
  // Skipped: /workstream not yet deployed — see Sprint 2026-04-01
  ...
});
```

---

### Category E: RBAC regression (P0)

**Symptom:** A test that asserts a role is **blocked** from a route instead passes — the blocked role can access a page they should not be able to reach. For example, `demo.viewer` successfully loads `/calibration`.

**Action:** This is a real security bug. Do not skip or retry. Investigate immediately:

1. Check `middleware.ts` for the route's auth guard
2. Check the page's `requireRole()` call in the server component or layout
3. Check the `ROLE_HIERARCHY` values in `lib/auth.ts` for drift

File a P0 issue. The failing assertion in the test is the intended behavior — the code is wrong, not the test.

---

### Category F: API contract failure

**Symptom:** `api-smoke.spec.ts` tests that assert specific JSON fields (e.g., `conflict_type` on SOD responses, `cutoverDate` on releases) fail because the field is absent or renamed.

**Action:** Check the API route handler for the endpoint and confirm the field name matches what the test expects. If the schema column was renamed in a migration, update both the API serializer and the test assertion together.

---

## Step 7 — Produce a Test Report

After running the full suite, produce a structured report:

```
Run all Vitest unit tests and all Playwright E2E tests.
For each failure:
1. State the spec file and test name
2. State the failure category (A–F from CLAUDE_CODE_TEST_RUNNER.md)
3. State the likely root cause (one sentence)
4. Propose the fix

Output a markdown table with columns:
File | Test Name | Category | Status | Root Cause | Proposed Fix
```
